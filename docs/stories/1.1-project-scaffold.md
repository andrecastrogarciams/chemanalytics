## Story

**As a** equipe técnica interna,  
**I want** uma base inicial do projeto com backend, frontend e configuração local padronizada,  
**so that** o MVP possa evoluir de forma incremental e validável desde o início.

## Acceptance Criteria

1. Existe estrutura inicial coerente com a spec para `backend/`, `frontend/`, `scripts/` e `docs/`.
2. O projeto possui arquivos base de configuração local e `.env.example` documentado.
3. Há um comando ou instrução CLI clara para subir backend e frontend em ambiente local.
4. O scaffold respeita o princípio `CLI First`, permitindo validar setup sem depender de UI pronta.
5. A story registra os arquivos principais criados/alterados na `File List`.
6. O scaffold deixa explícita a estratégia de separação entre backend Django REST e frontend React/Vite.
7. O setup inicial documenta as lacunas conhecidas de documentação referenciada no `core-config`, sem bloquear o bootstrap.

## Implementation Guidance

- Estrutura mínima esperada:
  - `backend/` para projeto Django e apps iniciais
  - `frontend/` para aplicação React/Vite
  - `scripts/` para utilitários operacionais e bootstrap
  - `docs/` para operação local e decisões do projeto
- Artefatos mínimos esperados:
  - `.env.example`
  - `backend/requirements*.txt` ou equivalente escolhido
  - `frontend/package.json`
  - documentação curta de bootstrap local
- Critério pragmático:
  - o desenvolvedor deve conseguir clonar, configurar ambiente e iniciar a base do projeto sem inferências externas.

## Risks / Open Questions

- Dependências de Oracle em Windows podem exigir toolchain C/C++ para build local quando usadas fora de wheel compatível.
- O projeto ainda não tem workflow de QA totalmente aderente ao formato atual flat de `docs/stories/`.
- A stack está definida na spec, mas os padrões finais de versionamento ainda podem ser refinados nas próximas stories.
